## 2. 项目简介

Atoms Demo 是一个由真实 AI Agent 驱动的代码生成工作台。用户可以用自然语言描述想要构建或修改的应用，Agent 会在隔离的项目环境中规划任务、执行命令、创建或修改文件，并把过程实时展示在对话、文件、终端和应用预览区域中。

本项目不是静态界面复刻，也没有使用预设脚本或模拟事件替代核心流程。当前版本已经形成以下真实闭环：

1. 选择或创建游客身份；
2. 输入自然语言需求并创建项目；
3. Agent 真实理解需求、规划任务并执行代码生成；
4. 页面实时展示 Agent 回复、Todo 进度、文件变更和终端输出；
5. 用户可以浏览代码文件，并在网页项目中直接查看运行结果；
6. 用户可继续发送修改要求，消息按顺序排队执行；
7. 对话、任务、文件、项目和版本信息持久化保存，刷新或重新打开后可恢复；
8. 在基础流程之外，进一步实现了 Todo 规划、结构化需求研发和代码版本管理能力。

## 4. 基础功能完成情况

### 4.1 游客与项目入口

- 系统提供 `default` 默认游客，也支持创建自定义游客。
- 游客名称具有格式、长度、重名和数量校验。
- 浏览器记住最近选择的游客，不同标签页的当前游客相互隔离。
- 用户可以从首页输入需求创建项目，也可以从“我的项目”查看历史项目。
- 项目列表支持分页、空状态、错误重试、最近更新时间和预览缩略图。
- 桌面侧栏支持折叠；窄屏使用移动导航抽屉。

### 4.2 自然语言创建与持续修改

- 首页输入有效需求后，系统自动创建项目并启动真实 Agent 任务。
- 支持 PNG、JPEG 图片选择与粘贴；单张图片最大 10 MB。
- 图片采用逐文件上传，可独立显示进度、失败、重试和移除，不限制单条消息的图片张数。
- 项目内可以继续发送修改要求。
- 同一项目任意时刻只执行一个任务，后续消息严格按接收顺序排队。
- 用户可以取消排队消息，也可以二次确认后停止正在运行的任务。
- 失败、停止和超时会保留已经产生的文件变化，避免丢失现场。

### 4.3 实时对话与执行过程

- Agent 面向用户的回复采用流式增量展示。
- 文件写入、终端命令、通用工具和工作流变化以结构化事件展示。
- 同一次文件写入原位更新，避免产生重复事件卡片。
- 对话使用统一的有序 Item 协议保存，重连后能够补齐缺失内容且不重复。
- 同一项目在多个标签页打开时，消息、事件、文件和任务状态实时同步。
- 用户主动上滚后不会被强制拉回底部，回到最新活动后恢复自动跟随。
- Markdown 内容经过安全渲染，不执行 Agent 返回的 HTML 或脚本。

### 4.4 文件、终端与应用预览

- 文件树支持按需展开、实时新增、修改、删除和重命名状态。
- 文本代码使用共享只读查看器，并提供受控语法高亮。
- 图片文件可以直接查看；过大文件和二进制文件使用安全降级展示。
- 终端区域展示真实命令、输出、状态和截断提示。
- 网页项目提供应用预览、刷新、路径导航、失败提示和重试。
- 应用预览运行在不授予 `allow-same-origin` 的 sandbox iframe 中，无法读取 Atoms Demo 主应用状态或浏览器存储。
- 非网页项目仍可以通过代码和终端完成查看，不伪装成可运行网页。

### 4.5 持久化、恢复与响应式体验

- 游客、项目、消息队列、对话、执行事件和工作流数据保存在 SQLite。
- 项目代码、附件、Agent session 和 Git 数据保存到持久化工作磁盘。
- 页面刷新、实时连接断开和服务重启后，可从服务端权威状态恢复。
- 后台任务完成、失败或取消时提供站内通知，可直接回到对应项目。
- 桌面端使用对话区与工作台双区布局，并支持拖拽调整比例或收起对话区。
- 移动端使用对话 / 工作台单区切换。
- 已验收 390、768、1024 和 1440 像素宽度，无页面级横向溢出。

## 5. 额外增强一：Todo 任务规划与进度跟踪

为提升长任务的可理解性，我增加了 Agent Todo 规划能力。

- Agent 处理多步骤任务时，先生成完整 Todo 列表，再开始执行。
- Todo 支持 `pending`、`in_progress`、`completed` 三种状态。
- 每次状态变化都会更新完整计划，并通过统一实时协议同步到页面。
- 输入框上方默认只显示紧凑的“第 X / Y 步”“已完成 X / Y”或“已停止 X / Y”。
- 用户悬停或键盘聚焦后，可查看完整任务列表和每一步状态。
- Todo 与所属任务轮次绑定，不会把上一轮计划错误展示为当前计划。
- 历史加载、刷新、断线重连、失败和取消后，Todo 状态仍能正确恢复。
- Todo 更新保留真实事件顺序，但不会以冗长的普通工具卡片干扰对话。

这个增强解决的不只是“展示一张清单”，而是让用户能持续理解 Agent 当前在做什么、已经完成什么、下一步是什么，以及任务是否正常结束。

## 6. 额外增强二：需求规划与版本管理

在基础的“输入需求后直接改代码”模式之外，我增加了更适合复杂需求的结构化研发模式，并将两种模式统一在工作项与版本体系下。

### 6.1 两种工作模式

#### 直接编码

适合范围清晰、希望快速得到结果的修改：

1. 从当前正式版本创建独立工作分支；
2. Agent 直接修改代码，可进行多轮对话；
3. 每轮有效变化形成内部检查点；
4. 用户确认发布前执行测试、类型检查、构建、敏感内容和预览检查；
5. 发布成功后形成不可变的正式版本。

#### 按需求研发

适合目标复杂、需要先澄清边界的需求：

1. 需求讨论：Agent 逐题访谈并持续整理需求文档；
2. 需求确认：检查目标、范围、非目标、依赖、待确认项和原子需求结构；
3. 技术设计：结合现状生成技术方案；
4. 技术确认：确认方案后才允许进入开发；
5. 开发：在独立工作分支实现，已确认需求和技术方案保持只读；
6. 测试准入：检查报告结构与测试条件；
7. 测试：运行已有测试、类型检查、构建和逐项验收；
8. 待上线：冻结候选工作区并提供候选预览；
9. 发布：用户二次确认后形成正式版本。

需求、技术、开发、测试和发布阶段有明确写入权限，Agent 不能在后续阶段静默修改已确认需求，也不能绕过测试直接发布。

### 6.2 工作项隔离

- 每个结构化需求或直接编码任务都是一个独立工作项。
- 工作项拥有独立对话、终端记录、Agent session、工作分支和审计事件。
- 每个项目同一时间最多有一个活动工作项，避免并发修改同一工作区。
- 发布或放弃后工作项归档；新任务创建新的工作项，不污染历史上下文。
- 失败、停止或服务重启后保留现场，恢复时先读取工作区、基线 Diff、Todo 和最近检查点。

### 6.3 项目级 Git 与正式版本

- 每个项目初始化独立 Git 仓库。
- Git 元数据位于 Agent 不可写的管理目录，commit、branch、merge 和 tag 仅由平台执行。
- 正式版本使用 `V1、V2……`，对应不可变 tag `code/v1、code/v2……`。
- 内部检查点不冒充正式版本；只有用户确认发布后才产生正式版本。
- 发布前复检版本内容边界，排除依赖、缓存、构建产物、日志、运行状态和秘密。
- 拒绝 `.gitmodules`、submodule、嵌套 Git、越界链接、特殊文件和超限二进制文件进入版本。
- 发布过程具备补偿逻辑；预检、Git 操作、预览或数据库写入失败时，不占用版本号，并尽量恢复发布前状态。

### 6.4 版本与工作记录查看

项目页提供三个版本入口：

- **当前工作**：查看活动工作项、阶段、文档、变化、测试和发布状态；
- **正式版本**：查看版本列表、来源、标题、摘要、确认人、commit、tag、文件统计和逐文件 Diff；
- **工作记录**：查看已发布或已放弃的历史工作、对话、终端、原因和时间线。

历史版本文件直接从 Git tree/blob 读取，不 checkout 或覆盖当前工作区。版本、页签和文件路径可以写入深链接，刷新后仍能恢复查看位置。

## 8. 工程思维与关键取舍

### 8.1 先建立真实最小闭环

最优先完成的是“自然语言需求 → Agent 执行 → 文件变化 → 网页预览 → 持续修改”的真实链路。界面动画和装饰性功能不能替代真实执行，因此核心流程没有使用模拟数据。

### 8.2 用稳定 ID 隔离数据

游客名称和项目名称只用于展示，资源隔离使用稳定的游客 ID、项目 ID 和工作项 ID，避免重名、改名或特殊字符破坏目录与数据关系。

### 8.3 把实时事件统一成一种协议

Agent 文字、命令、文件变化、Todo 和状态变化都进入有序 Conversation Item 协议。页面实时展示、历史加载和断线恢复共享同一数据模型，避免出现“实时看到了，但刷新后丢失”或顺序错乱。

### 8.4 将 Agent 权限限制在当前项目

Agent 可以在当前项目内自动执行普通文件和终端操作，但不能读取其他项目、宿主文件、云元数据或预置密钥。所有操作同时向用户展示，兼顾执行效率、隔离性与可追踪性。

### 8.5 区分内部检查点与正式版本

开发过程需要频繁保存，但频繁 commit 不应污染用户看到的版本历史。因此内部检查点用于恢复和审计，正式版本只在质量检查和用户确认后产生。

### 8.6 用门禁控制复杂工作流

结构化需求模式不是只增加几个状态标签，而是由服务端校验阶段、revision、幂等键、文档结构、测试报告和发布条件。按钮与自然语言操作共用同一套规则，避免只在前端限制而被绕过。
