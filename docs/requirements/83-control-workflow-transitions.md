# 控制工作流阶段流转

## 目标

用户可以通过按钮或自然语言可靠地推进、退回、停止或放弃工作，同时避免误触造成状态变化。

## 需求

1. 结构化需求状态为：需求讨论中、技术设计中、技术方案待确认、开发中、测试中、待上线、已上线、已放弃；“需求待确认”仅用于兼容升级前已经停留在该阶段的工作项。
2. 运行中、已停止、执行失败和等待重试作为执行状态附加在当前阶段，不创建额外流程阶段。
3. 输入框上方展示需求编号、当前阶段、执行状态和当前主要操作。
4. 低频或危险操作放入“更多”菜单。
5. 按钮和自然语言使用同一动作识别与执行机制。
6. 确认需求、确认方案并开始开发、确认上线、新建需求、退回阶段、放弃需求、放弃未发布改动和停止任务需要二次确认。
7. 继续执行、重试当前阶段、普通对话和查看操作不需要二次确认。
8. 二次确认只确认操作后果，不实现内容哈希失效机制。
9. 存在运行中或排队消息时禁止阶段流转，“停止”除外。
10. 不允许跳过阶段向前推进；已上线工作项不能退回。
11. 对话中插入只读阶段事件卡片。
12. 用户确认需求即授权未单独回答的事项采用推荐默认方案；平台校验当前编号需求包、创建检查点并直接进入技术设计，不再启动重复整理需求的智能体轮次。仅当编号需求包尚未完整落盘时，允许创建一次原地补全轮次。

## 验收标准

- 自然语言与按钮产生相同的待确认动作。
- 取消确认后状态和文件均不变化。
- 越级、重复和并发流转被服务端拒绝。
- 退回原因、触发方式和结果可追溯。

## 不在本任务范围内

- 各阶段具体工作内容。
- Git 发布实现。

## 独立性与原子性检查

- **单一职责**：只负责工作流动作、门禁和交互确认。
- **可独立验证**：覆盖前进、退回、停止、重试、放弃和并发操作即可验证。
- **结论**：虽然包含多个动作，但它们共享同一个不可拆分的状态机与确认协议。
