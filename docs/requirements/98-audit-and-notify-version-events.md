# 审计并通知版本事件

## 目标

重要工作流与版本操作可追溯，等待用户处理或后台完成时得到适量通知。

## 需求

1. 追加式审计事件记录工作项创建、阶段流转、退回原因、触发来源、二次确认、检查点、停止、重试、失败、发布尝试和放弃。
2. 完整审计记录用于一致性恢复和排障，不允许普通用户修改。
3. 产品界面只展示简化时间线。
4. 站内通知覆盖：需求待确认、技术方案待确认、开发完成、测试通过待上线、开发/测试/发布失败和版本发布成功。
5. 普通文档增量更新和内部检查点不发送通知。
6. 通知深链接到对应当前工作、工作记录或版本详情。
7. 正式版本详情记录发起及确认操作的游客身份。

## 验收标准

- 每个关键动作都有顺序稳定的审计事件。
- 后台状态变化产生正确通知，重复事件不产生重复通知。
- 点击通知准确定位目标。

## 不在本任务范围内

- 邮件、短信或第三方通知。
- 产品分析埋点。

## 独立性与原子性检查

- **单一职责**：只负责版本域审计和站内通知。
- **可独立验证**：触发全部关键事件并检查记录与通知即可验证。
- **结论**：任务足够独立且原子。
