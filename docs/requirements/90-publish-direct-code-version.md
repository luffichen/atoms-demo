# 发布直接编码代码版本

## 目标

用户将一个直接编码工作项的累计改动经过轻量验证后发布为正式代码版本。

## 需求

1. 成功编码只产生内部检查点，不自动占用正式版本号。
2. “版本”的“当前工作”展示累计对话、Diff、验证状态和“发布代码版本”。
3. 系统根据当前工作项对话和 Diff 生成标题与摘要，用户可在确认框中修改。
4. 每次发布尝试均需二次确认；失败后再次尝试必须重新确认。
5. 发布前运行已有测试、类型检查、Lint、构建、敏感文件检查和预览启动检查。
6. 没有自动化脚本时允许发布，但必须明确标记并完成最低文件、配置和预览验证。
7. 验证失败时阻止发布，不支持忽略失败强制发布。
8. 直接编码使用独立工作分支和 `--no-ff` 发布合并，版本号与结构化需求共享 `V1、V2…` 序列。
9. `docs/releases/Vn.md` 记录来源为“直接编码”、对话范围、摘要、Diff、轻量验证和发布结果。
10. 直接编码版本不伪造需求编号、需求文档或技术方案。

## 验收标准

- 多轮改动只在用户确认后形成一个正式版本。
- 自动生成内容可在确认前编辑，发布后不可变。
- 无测试项目的例外被明确披露。
- 发布后工作项归档，继续修改需要新工作项。

## 不在本任务范围内

- 选择部分文件发布。
- 强制跳过验证。
- 结构化需求文档。

## 独立性与原子性检查

- **单一职责**：只负责直接编码工作项的验证与正式发布。
- **可独立验证**：覆盖有测试、无测试、失败重试和成功发布即可验证。
- **结论**：任务足够独立且原子。
