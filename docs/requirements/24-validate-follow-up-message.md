# 校验项目后续消息

## 目标

用户在项目中继续提出修改要求时，系统只接受内容完整且长度合规的消息。

## 需求

1. 后续消息必须包含至少一个非空白文字。
2. 只包含空格、制表符或换行的内容无效。
3. 单条消息最多包含 10,000 个可见字符。
4. 图片只能补充文字，不能单独发送。
5. `Enter` 提交消息，`Shift + Enter` 插入换行。
6. 文字无效、超长或图片处于失败状态时，按钮和键盘提交均不可用。
7. 图片类型、单文件大小和发送失败规则与首页一致；图片张数及消息图片总大小不限。
8. 首版不展示语音输入或智能体生成的后续建议按钮。

## 验收标准

- 空内容、纯空白和仅图片消息均不能发送。
- 1 至 10,000 个可见字符可以发送，超过限制不能发送并显示原因。
- `Enter` 与 `Shift + Enter` 分别执行提交和换行。
- 无效消息不会进入队列。
- 图片附件的全部已确认边界在项目输入区同样生效。

## 不在本任务范围内

- 把有效消息加入队列。
- 执行智能体任务。
- 展示历史对话。

## 独立性与原子性检查

- **单一职责**：只判断项目内一条后续消息能否提交。
- **可独立验证**：使用文字长度、空白和附件组合即可覆盖。
- **结论**：任务足够独立且原子，无需继续拆分。
