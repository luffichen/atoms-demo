# 处理服务重启中断

## 目标

服务重启导致当前任务无法继续时，项目进入明确、可继续的状态，而不是永久显示执行中。

## 需求

1. 首版不要求从中断点继续执行正在运行的消息。
2. 服务恢复后，将重启时执行中的消息标记为“失败”。
3. 失败原因显示“服务重启导致中断”。
4. 中断前已经产生的文件变化继续保留，不自动回滚。
5. 已排队消息继续保留。
6. 当前消息标记失败后，自动开始下一条排队消息。
7. 没有后续消息时，项目进入空闲并允许用户继续发送。

## 验收标准

- 执行中重启服务后，原消息不会永久停留在执行中。
- 恢复后显示约定失败原因，已有文件仍可查看。
- 后续排队消息按原顺序继续执行。
- 没有队列时可以提交新的修复要求。

## 不在本任务范围内

- 执行中断点续跑。
- 自动回滚文件。
- 普通网络断线重连。

## 独立性与原子性检查

- **单一职责**：只把服务重启留下的执行中状态恢复为可继续状态。
- **可独立验证**：在有、无后续队列两种情况下重启即可验证。
- **结论**：任务足够独立且原子，无需继续拆分。
