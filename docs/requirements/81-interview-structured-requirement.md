# 逐题访谈结构化需求

## 目标

智能体在修改代码前与用户逐项消除需求歧义，形成共享理解。

## 需求

1. 需求讨论阶段固定注入以下提示：

   ```text
   围绕尚未明确且会影响用户可见行为或验收结果的产品决策进行访谈，逐项解决依赖关系，直到目标、范围和验收标准达成共同理解；每个问题给出推荐答案。
   一次只提出一个问题。
   能通过代码库、现有文档或项目约定回答的问题，先自行探索，不要反问用户。
   需求讨论只解决产品层决策；实现层决策留到技术设计阶段。
   ```

2. 智能体一次只能提出一个问题，并为该问题给出推荐答案。
3. 能通过代码、现有文档或项目约定确认的内容必须先自行探索。
4. 用户可连续排队补充消息；智能体在提出下一问题前将其作为当前问题的补充统一理解。
5. 补充消息冲突时只询问一个具体冲突点。
6. 访谈覆盖产品层的目标用户、使用场景、功能与内容范围、用户流程、可见交互、业务状态、异常体验、权限规则、产品兼容性、验收标准和非目标。
7. 技术栈、框架或库选型、渲染方式、模板引擎、模块划分、API、代码或数据结构、存储、CSS 工程方案、部署和测试工具属于技术设计阶段，需求讨论阶段不得询问或自行展开。
8. 用户主动给出的框架、数据库、部署环境等技术信息只记录为已确认的实现约束，不沿其继续追问相邻技术选型。
9. 提问前必须区分产品决策与实现决策；只有实质影响用户可见行为或验收结果、且无法合理推断的产品决策才允许询问。
10. 用户明确授权采用推荐方案或最优默认值时，智能体直接采用合理默认值并记录关键假设，不再继续提问。
11. 需求阶段的待办列表和需求文档保持产品视角，不得包含自行推导的架构、模块、API 或数据模型设计。
12. 需求讨论阶段代码只读，不允许提前实现、安装依赖或执行产生业务改动的命令。

## 验收标准

- 智能体不会在一条消息中抛出多个问题。
- 代码可回答的问题不会反问用户。
- 智能体不会在需求阶段询问 SSR/SPA、模板引擎、模块、API、数据结构等实现方式。
- 用户指定 NestJS、Mock 数据等实现约束时，智能体直接记录，不继续追问相邻技术选型。
- 用户授权采用最优默认值后，智能体不再继续访谈。
- 需求草稿只包含产品需求与用户明确给出的实现约束，不包含自行推导的技术方案。
- 多条排队补充被合并理解，不被机械当作后续问题答案。
- 访谈期间业务代码保持不变。

## 不在本任务范围内

- 需求文档格式。
- 技术设计访谈。
- 阶段确认。

## 独立性与原子性检查

- **单一职责**：只负责结构化需求的访谈行为。
- **可独立验证**：提供包含已知代码事实、冲突补充和未决产品分支的需求即可验证。
- **结论**：任务足够独立且原子。
