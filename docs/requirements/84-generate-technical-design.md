# 生成并确认技术方案

## 目标

已确认需求转化为基于现有代码、可实施且可测试的技术方案。

## 需求

1. 用户确认需求后，系统先进入“需求待确认”，自动创建一轮需求文档整理任务，基于本工作项前面的完整对话生成带当前需求编号的正式需求文档。
2. 只有需求文档成功写入 `docs/requirements/Rxxx-*/`、通过存在性校验并创建需求检查点后，系统才能切换到技术设计阶段。
3. 进入技术设计阶段后，平台自动创建并启动技术设计任务；用户无需再次发送指令。
4. 智能体先读取刚生成的已确认需求文档并探索当前代码版本。
5. 技术阶段只询问重大产品影响、兼容性、迁移和风险取舍；纯实现细节自行遵循现有项目惯例。
6. 技术文档位于 `docs/technical/Rxxx-feature.md`。
7. 文档必须包含背景与需求引用、现状分析、总体方案、数据模型、接口与事件、前端交互、安全边界、兼容性、实施步骤、测试计划、上线回滚、风险和非目标。
8. 不适用章节必须说明原因，不能静默遗漏。
9. 跨需求长期有效的技术决策同步到 `docs/technical-decisions.md`。
10. 技术文档无待确认项后进入“技术方案待确认”。
11. “确认方案并开始开发”二次确认后锁定方案、创建内部检查点并开始开发。

## 验收标准

- 技术方案引用明确需求并包含完整强制章节。
- 完成“确认需求”二次确认后先出现需求文档整理轮次；正式需求文档生成成功后才出现技术设计轮次。
- 需求文档缺失或生成失败时停留在“需求待确认”，不得进入技术设计。
- 智能体先探索代码，避免反问可自行确定的实现细节。
- 确认前不修改业务代码；确认后自动进入开发。

## 不在本任务范围内

- 代码实现。
- 测试执行。
- 正式版本发布。

## 独立性与原子性检查

- **单一职责**：只负责技术方案的产生、讨论和确认。
- **可独立验证**：确认一份需求并检查技术访谈、文档和确认门禁即可验证。
- **结论**：任务足够独立且原子。
