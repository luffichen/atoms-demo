# 实时展示文件写入

## 目标

智能体分批写入文件时，用户能够持续看到同一次操作的最新进度和已经写入的内容。

## 需求

1. 同一次文件写入在左侧只产生一张事件卡片。
2. 分批写入时，在原卡片上更新进度和状态，不新增重复卡片。
3. 文件操作开始时，如果用户没有手动锁定其他查看器，立即打开“文件”模式并选中目标文件。
   - 用户仅进入“文件”模式等待生成，不视为锁定具体文件；后续文件写入仍应自动选中并展示。
   - 用户明确选择另一个文件后，才保留该文件的阅读位置，不再自动抢占。
4. 每次原始 `write` 前，智能体先调用不落盘的 `file_create(path)` UI 预告工具；服务端在预告路径完整后通过同名 UI 事件创建文件树占位并自动打开空文件。
5. 随后的原始 `write` 内容参数通过 UI 专用 `file_append` 事件增量展示，不等待完整工具参数生成或物理写入完成；即使模型在 `write` JSON 中先生成 `content`、最后才生成 `path`，也必须使用已预告路径持续展示。
6. `file_create`、`file_append` 不写磁盘、不创建对话卡片，也不改变原始 `write` 工具的一次性落盘语义。
7. `write` 工具参数完整后，持久化 `file_change` item 及其完整快照，供历史和重连使用。
8. 不能预先得到完整目标内容的操作应立即展示磁盘当前内容，并在操作完成后刷新。
9. 写入成功后使用磁盘内容校准查看器。
10. 写入完成、失败或取消后，卡片显示对应终态。
11. 已显示的内容在操作失败或取消后继续可见。
12. 工作区标签栏右侧提供统一的“自动跟随”开关，不放在文件查看器内部；该开关同时服务文件和终端，新 turn 开始时默认开启。
13. 自动跟随开启时，`file_append` 到达后滚动到最新内容；创建第二个或后续文件时自动切换到最新文件。
14. 用户在文件或终端中主动向上滚动超过 80px，或手动选择其他文件时关闭自动跟随，避免打断阅读。
15. 自动跟随关闭后不会因为用户回到底部而自行恢复；用户可以通过开关手动重新开启。
16. 用户重新开启时，切换到最近发生实时活动的文件或终端；如果仍有流式文件正在生成，立即打开最新流式文件并继续跟随。

## 验收标准

- 一次分多批写入的操作始终只有一张事件卡片。
- 新建文件事件开始后，无需点击卡片即可看到目标路径和待写入内容。
- 大型 `write` 工具参数尚未生成完成时，文件树已出现预告目标文件，查看器内容随参数流持续增长，包括 `content` 位于 `path` 之前的参数顺序。
- UI 增量期间磁盘文件不存在或仍保持旧内容；仅原始 `write` 执行时更新磁盘。
- 自动跟随开启时持续跟随新增内容和后续新文件，用户上滚后关闭并保持当前阅读位置。
- 写入成功后，查看器内容与磁盘文件一致。
- 完成、失败和取消状态正确更新在原卡片。
- 失败或取消不会让此前展示的内容消失。

## 不在本任务范围内

- 文件树状态标记。
- 手动打开其他文件。

## 独立性与原子性检查

- **单一职责**：只呈现一次文件写入从开始到终态的实时变化。
- **可独立验证**：使用一个分批写入事件即可覆盖核心行为。
- **结论**：任务足够独立且原子，无需继续拆分。
