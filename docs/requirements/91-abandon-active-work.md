# 放弃活动工作

## 目标

用户可以安全终止不再继续的工作，并明确区分结构化需求归档和直接编码改动清除。

## 需求

1. “放弃需求”和“放弃未发布改动”同时支持按钮与自然语言，并要求二次确认。
2. 结构化需求放弃前将当前受跟踪内容创建为“放弃快照”内部提交。
3. 已放弃需求保留需求编号、对话、session、文档、终端、检查点、放弃原因和时间。
4. 已放弃需求不合入 `main`、不创建正式版本、不能恢复或继续。
5. 已放弃需求的分支保留。
6. 直接编码放弃后工作区恢复当前正式版本；没有正式版本时恢复初始化基线。
7. 放弃的直接编码代码不提供历史代码或恢复能力，但工作项对话与 session 作为工作记录保留。
8. 放弃后项目可以创建新工作项。

## 验收标准

- 取消二次确认不会改变工作区或状态。
- 已放弃需求可只读查看完整快照，但不能继续执行。
- 已放弃直接编码改动从工作区清除且无法恢复。
- 正式版本不受任何放弃操作影响。

## 不在本任务范围内

- 删除已放弃记录。
- 恢复已放弃代码。
- 清理 Git 存储。

## 独立性与原子性检查

- **单一职责**：只负责终止活动工作及其归档/清除语义。
- **可独立验证**：分别放弃两类工作项并检查代码、历史和新工作入口即可验证。
- **结论**：两类工作共享同一终止入口，但其数据保留差异不可省略。
