# 实现结构化需求

## 目标

智能体按照已确认需求和技术方案完成代码实现，并留下稳定的阶段检查点。

## 需求

1. 开发从当前正式版本创建的独立工作分支执行。
2. 智能体以已确认需求和技术方案作为权威输入，不要求用户复制内容。
3. 代码实现完成后自动创建“实现完成”内部 Git 检查点。
4. 内部检查点固定包含：需求确认、技术方案确认、实现完成、测试通过和发布合并。
5. 阶段内每条消息不单独产生碎片 commit。
6. 用户提出实现缺陷时可以在当前范围内修复。
7. 用户提出改变功能、流程、数据、权限或验收标准的要求时，暂停开发并提出退回相应阶段的待确认动作。
8. 用户主动停止或异常失败时停留在开发阶段，保留未提交现场，等待手动继续或重试。
9. 重试前读取当前文件、Diff、待办和最近检查点，再继续未完成步骤。

## 验收标准

- 实现严格对应确认文档。
- 需求变化不会被静默吸收到代码中。
- 停止、失败和服务重启后现场保留且不自动续跑。
- 实现完成检查点只在实现阶段确实完成后产生。

## 不在本任务范围内

- 测试门禁。
- 正式上线。

## 独立性与原子性检查

- **单一职责**：只负责结构化需求的代码实现和实现阶段恢复。
- **可独立验证**：覆盖正常实现、范围变化、停止和失败重试即可验证。
- **结论**：任务足够独立且原子。
