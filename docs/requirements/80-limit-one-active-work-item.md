# 限制单个活动工作项

## 目标

项目始终只有一个可写工作区、一个活动对话和一个活动 session，避免分支、预览与事件冲突。

## 需求

1. 每个项目同一时刻最多存在一个未归档工作项。
2. 活动工作项可以是结构化需求或直接编码，不能并存。
3. 活动工作项发布或放弃前禁止创建另一工作项。
4. 结构化需求活动期间，历史直接编码对话只读。
5. 开始结构化需求前，直接编码工作区必须与当前正式版本一致。
6. 存在未发布直接编码改动时，提示用户先发布或放弃改动。
7. 不实现需求队列、多工作树或并行分支研发。

## 验收标准

- 活动需求期间无法启动直接编码工作。
- 未发布直接编码改动存在时无法创建需求。
- 工作项归档后可以立即开始下一项工作。

## 不在本任务范围内

- 多需求排队。
- Git worktree。
- 跨项目并发。

## 独立性与原子性检查

- **单一职责**：只负责项目内工作项互斥。
- **可独立验证**：在各类活动和脏工作区状态下尝试创建新工作即可验证。
- **结论**：任务足够独立且原子。
