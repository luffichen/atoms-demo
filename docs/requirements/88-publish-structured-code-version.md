# 发布结构化需求代码版本

## 目标

经过确认、实现、测试和候选预览的需求以不可变正式代码版本发布。

## 需求

1. 可预览网页在待上线阶段展示“待上线预览 · Rxxx”。
2. 预览启动失败时不能上线；非网页项目跳过预览门禁。
3. 用户通过按钮或自然语言发起“确认上线”，二次确认后执行发布。
4. 发布使用 `--no-ff` 将需求分支合入 `main`，保留内部检查点历史。
5. 项目内正式版本按成功顺序编号 `V1、V2…`，Git tag 为 `code/v1、code/v2…`。
6. 版本号只在发布成功时占用，不复用、不修改。
7. 发布必须全有或全无：门禁复检、合并、tag、版本记录、当前版本切换、正式预览和发布记录全部成功后才可见。
8. 失败时停留在待上线，由用户手动重试。
9. `docs/releases/Vn.md` 记录需求、基线、提交、tag、时间、确认身份、变化、文档、测试、预览和已知限制。
10. 发布成功后删除工作分支引用、保留提交历史并归档工作项。
11. 首版上线只表示项目内正式版本和预览切换，不部署外部生产环境。

## 验收标准

- 成功发布生成连续版本号、tag、不可变记录和正式预览。
- 任一步失败不会暴露半成品版本。
- 发布后需求对话只读，新修改必须创建新工作项。

## 不在本任务范围内

- 外部部署。
- 应用版本。
- 正式版本删除、重命名或回滚。

## 独立性与原子性检查

- **单一职责**：只负责结构化需求的原子正式发布。
- **可独立验证**：覆盖成功和各关键步骤失败即可验证。
- **结论**：发布事务必须整体保持原子，不能继续拆分。
