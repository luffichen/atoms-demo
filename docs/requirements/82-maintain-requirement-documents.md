# 维护需求文档

## 目标

结构化需求讨论的已确认结论实时沉淀为仓库内可版本化的 Markdown 文档。

## 需求

1. 文档位于 `docs/requirements/Rxxx-feature/`。
2. `README.md` 记录总体目标、范围、非目标、依赖和原子需求索引。
3. 复杂需求拆分为多个可独立实现、独立验收的原子需求文档，但共同形成一个工作项和一个正式版本。
4. 从需求讨论第一轮起，智能体直接增量维护最终编号目录，不在 `docs/requirements` 根目录另建一套重复草稿。
5. 每个问题必须给出推荐答案；尚未得到用户回答的事项先以推荐方案或合理默认值写成当前结论，并标注来源为“默认采用”。
6. 冲突结论替换旧结论，并维护简短决策变更记录。
7. 首版不提供 Markdown 直接编辑器，也不提供逐轮文档版本浏览；修改通过需求对话完成。
8. 被需求文档引用的对话图片在确认时复制到需求包 `assets/` 并进入 Git。
9. 未引用图片只保留为对话附件。

## 验收标准

- 访谈结论可以在“版本”查看器中实时看到草稿更新。
- 原子需求有索引且每项包含目标、需求、验收标准、非目标和原子性检查。
- 用户确认需求时，未单独回答的事项沿用文档中的默认方案，不触发第二次全文生成。
- 若智能体异常地尚未完成首次落盘，确认后只创建一次补全轮次；已有需求包必须原地更新，不得新建第二套目录。
- 用户确认、默认采用和决策变化的来源清晰可辨。
- 被引用图片在历史版本中仍可访问。

## 不在本任务范围内

- 技术方案文档。
- 正式版本发布记录。
- 文档直接编辑。

## 独立性与原子性检查

- **单一职责**：只负责需求文档的结构和增量维护。
- **可独立验证**：完成多轮访谈、修改结论并引用图片即可验证。
- **结论**：任务足够独立且原子。
