# 处理消息失败与超时

## 目标

单条任务失败或执行过久时，用户能看到明确结果，同时其他排队任务不会被永久阻塞。

## 需求

1. 普通执行错误时，消息状态变为“失败”并展示原因。
2. 单条消息最长执行 30 分钟。
3. 达到时限后停止该任务，状态变为“失败”，并提示“执行超时”。
4. 失败或超时前已经产生的项目文件变化继续保留。
5. 系统不自动回滚文件变化。
6. 失败或超时处理完成后，自动开始下一条排队消息。
7. 用户可以随后发送新的修复要求。

## 验收标准

- 制造普通执行错误后看到失败状态和可理解的原因。
- 任务达到 30 分钟时停止，并明确显示执行超时。
- 两类失败发生前写入的文件仍然存在。
- 有后续消息时自动继续执行下一条；没有时进入空闲。
- 失败不妨碍用户继续提交新的有效消息。

## 不在本任务范围内

- 自动重试失败任务。
- 自动回滚文件。
- 恢复任务断点。
- 用户主动停止任务。

## 独立性与原子性检查

- **单一职责**：只把非正常结束转为明确终态并释放队列。
- **可独立验证**：普通错误和超时共享相同队列恢复行为，可在一个任务中覆盖。
- **结论**：任务边界集中且可验证，无需继续拆分。
