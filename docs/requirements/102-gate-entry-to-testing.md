# 测试阶段准入检查

## 目标

结构化需求完成开发后，先通过可验证的测试准入检查和自动修复闭环，再进入正式测试阶段。

## 需求

1. 用户执行“开始测试”后，工作项先进入“测试准入”，不得直接进入“测试”。
2. 准入任务检查已确认需求文档、技术方案、实现完整性、自动化测试，以及项目实际提供的类型检查、Lint 和构建。
3. 检查失败时，智能体先在当前对话中明确失败项和计划采取的修复动作。
4. 属于已确认范围内的代码或测试问题由智能体自动修复，并重新执行受影响检查。
5. 失败暴露需求范围或技术方案缺陷时不得擅自修改已确认文档，工作项停留在测试准入并提示需要退回的阶段。
6. 全部检查通过后生成 `docs/test-reports/Rxxx-admission.json`；报告必须包含检查结论、命令、退出码、证据、发现的问题和修复记录。
7. 服务端校验准入报告及全部必需证据，通过后创建实现检查点，自动进入测试阶段并启动完整测试任务。
8. 准入任务异常或无法修复时停留在当前阶段，用户可使用“重新执行”再次触发同一闭环。
9. 智能体结束后若服务端报告校验仍失败，系统将具体原因自动发回同一会话并最多重试两轮；达到上限后才停留在失败状态等待用户处理。

## 验收标准

- 点击“开始测试”后的首个可见阶段是“测试准入”。
- 任一必需检查缺失、失败或命令退出码非零时，服务端不进入测试阶段。
- 可修复失败会先产生用户可见说明，随后自动修复并复检，无需用户再次确认。
- 只有机器可校验的准入报告通过后，系统才自动创建正式测试任务。
- 准入失败后工作项仍可退回开发阶段或手动重新执行。

## 不在本任务范围内

- 替代正式测试阶段的完整验收与测试报告。
- 自动修改已确认需求或技术方案。
- 对不同技术栈硬编码固定命令。

## 独立性与原子性检查

- **单一职责**：只负责开发完成到正式测试之间的准入门禁与自动修复。
- **可独立验证**：分别制造直接通过、可自动修复和不可修复三类结果即可验证。
- **结论**：准入检查、自动修复和阶段推进属于同一闭环，继续拆分会破坏门禁结果的原子性。
